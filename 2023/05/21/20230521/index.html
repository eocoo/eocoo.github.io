软件只是一段没有生命的静态代码，只有在物理硬件上运行之后才会被赋予“生命”，操作系统则负责管理硬件资源，并在此基础上为软件提供所需的运行环境，这里尝试从CPU硬件架构和操作系统的角度来总结下对于并发问题的处理思路，并在此基础上更好的理解相关语言设计。

进程

进程Process是指计算机中已运行的程序，是系统进行资源分配的基本单位。每个进程都有自己的地址空间，包括text文本区域、data数据区域和stack堆栈，文本区域存储处理器执行的代码，数据区域存储变量和进程执行期间使用的动态分配的内存，堆栈区域存储着活动过程调用的指令和本地变量。

进程间通信

在操作系统中，每个进程都有自己独立的地址空间，也就是说进程的内存空间、代码段、数据段等资源都是私有的，其他进程无法直接访问，这种独立性和隔离性是操作系统设计的重要原则。这种设计提高了系统的稳定性和安全性，如果一个进程出现了错误，比如内存访问越界、非法指令等，只会影响到它自身，而不会波及其他进程，

但这种独立性也带来了一个问题，那就是进程之间的数据无法直接共享和交互。而在很多实际场景中，不同进程之间需要相互协作，这时就需要借助IPC机制来打破进程之间的隔离，实现进程间的数据传输和共享。

虽然每个进程的用户地址空间都是独立的，一般而言一个进程无法直接访问另一个进程的地址空间，但是内核空间是每个进程共享的，所以进程之间想要进行信息交换就必须通过内核。

  • 管道 Pipe

管道是Unix中最古老的进程间通信形式，它本质上是内核的一块缓存，有单向数据传输的特点，即数据只能从管道的一端写入，从另一端读出，并遵循FIFO先进先出的原则，就像一个阻塞队列，

对于管道两端的进程而言，管道就是一个文件（所以也被称为共享文件机制），但它不是普通的文件，不属于某种文件系统，只存在于内存中，然后对于管道文件的操作，会被内核转换成对缓冲区的操作。

  • 消息队列 Message Queue

管道方式虽然使用简单，但是效率比较低，不适合进程间频繁地交换数据，并且管道只能传输无格式的字节流。所以可以进一步使用消息队列机制，其本质是存放在内存中的消息链表，而消息就是用户自定义的数据结构，如果进程从消息队列中读取了某个消息，这个消息就会被从消息队列中删除。

消息队列提供了一种异步通信的方式，发送和接收进程不需要同时运行，而且消息可以按照类型进行分类，方便进程根据自身需求接收特定类型的消息。但是消息队列的效率相对较低，因为消息的发送和接收都需要进行系统调用，存在一定的开销，数据需要在内核态和用户态之间来回拷贝，并且消息队列的大小有限，如果消息队列满了，发送消息的操作会被阻塞。

  • 共享内存 Shared Memory

共享内存允许不相干的进程将同一段物理内存映射到它们各自的地址空间中，使得这些进程可以访问同一段物理内存。某个进程向共享内存写入数据后，其改动将立即影响到可以访问同一段共享内存的任何其他进程。

不同于消息队列频繁的系统调用，对于共享内存机制来说，仅在建立共享内存区域时需要系统调用，一旦建立共享内存，所有的访问都可作为常规内存访问，无需借助内核。这样，数据就不用在进程之间来回拷贝，所以这是最快的一种进程通信方式。

  • 信号量Semaphore和PV操作

要保证共享内存在任何时刻只有一个进程在访问（互斥），并且使得进程们能够按照某个特定顺序访问共享内存（同步），可以使用进程的同步与互斥机制，常见的比如信号量与PV操作。

信号量就是一个变量，其作用在于表征系统中某类资源的可用数量。当一个进程想要访问共享资源时，首先会检查信号量的值。如果信号量的值大于0，说明有可用的资源，进程可以将信号量的值减1，然后访问共享资源；如果信号量的值为0，说明资源已被占用，进程会被阻塞，直到信号量的值大于0。

P操作：执行P操作时，会先将信号量的值减1，这一操作对应进程申请占用一个系统资源。

V操作：执行V操作时，会将信号量的值加1，对应进程使用完资源后，将资源归还系统的操作。

  • 信号 Signal

信号和信号量是完全不同的两个概念，它是进程通信机制中唯一的异步通信机制，可以在任何时候发送信号给某个进程。通过发送指定信号来通知进程某个异步事件，以迫使进程执行信号处理程序，信号处理完毕后，被中断进程将恢复执行。

信号事件的来源主要有硬件和软件，硬件来源可以通过键盘输入某些组合键给进程发送信号，比如常见的Ctrl+C，软件来源可以通过kill系列命令给进程发送信号，比如kill -9让进程立即结束。

  • 套接字 Socket

套接字是一种基于网络协议栈的IPC机制，可以让跨网络与不同主机上的进程进行通信。通过“IP地址+端口号”定位网络中的进程，建立双向全双工的通信链路，数据可以双向传输，无需像管道那样区分读写端。

main函数

我们都知道main函数是代码的入口，但实际上main函数只是用户级代码的入口，它也是被其他函数调用的。比如在C语言中，由操作系统提供的启动例程_start加载初始化运行时，然后调用__libc_start_main执行main函数内容，最后结束时再通过系统调用exit(exit_code)返回退出码。在Java语言中，操作系统通过java命令来创建jvm进程，然后由jvm调用执行main函数，如果正常结束，JVM会调用System.exit(0)，否则调用 System.exit(非0)返回操作系统。

既然main函数是间接被操作系统调用的，那么main函数结束时应该要给操作系统返回相应的退出信息，而这个退出信息就是以退出码的形式作为返回值，一般以0表示执行成功，以非0表示执行出现错误（成功只有一种情况，而出现错误则有各种可能），可以通过echo $?查看进程最近一次的退出码。

exec函数

fork创建子进程后，子进程执行的是和父进程相同的程序（但有可能执行不同的代码分支），若想让子进程完全执行另一个程序，可以通过exec函数。当进程调用exec函数时，该进程的用户空间代码和数据会完全被新程序替换，并从新程序的启动例程开始执行。

exec函数如果调用成功，则加载指定的程序并从启动代码开始执行，不再返回。如果调用出错，则返回-1，也就是说只要返回了，就意味着调用失败。所以如果单独用exec没什么太大意义（它会把当前进程替换掉），通常的用法是与fork配合使用

wait等待是必要的，父进程需要通过进程等待的方式，回收子进程资源，获取其退出信息。如果子进程退出，父进程不读取子进程的退出信息，子进程就会变成僵尸进程，进而造成内存泄漏。进程一旦变成僵尸进程，那么就算是kill -9命令也无法将其杀死，因为谁也无法杀死一个已经死去的进程。

当父进程退出时，其所有的僵尸子进程会被系统的init进程（PID=1）收养，init进程会自动调用wait清理它们。所以如果父进程很快就退出了，那么影响不大，否则如果长期运行，则僵尸子进程会造成PID耗尽和内存泄漏，可以通过ps aux | grep Z命令查看。

线程

线程是进程的一个执行单元，是CPU调度的基本单位。一个进程可以包含多个线程，这些线程共享进程的资源，可以将进程比作一个独立的工作室，拥有自己的办公场地和设备，然后线程就是员工，它们共享场地和设备来完成进程的任务。与进程相比，线程具有更低的创建和销毁开销，以及更快的上下文切换速度，这使得线程在处理一些需要频繁创建和销毁执行单元的场景中，具有明显的优势。

但多线程编程中需要确保数据一致性和避免竞态条件，互斥锁是最常用的线程同步工具之一，在pthread库中，pthread_mutex_t类型的互斥锁用于保护临界区，确保同一时间只有一个线程能够进入临界区访问共享资源，其内核层面的实现利用了原子操作和futex。原子操作保证了CPU执行加锁和解锁操作时不可中断，比如一个线程尝试加锁的过程，需要先检查锁的状态，如果锁处于未被占用状态，再将其设置为已占用状态。

futex是一种用户空间和内核空间协作的同步机制，当一个线程发现锁已被占用时，它会通过futex系统调用进入内核态，将自己放入等待队列中，并将当前线程的状态设置为睡眠状态让出CPU资源。当持有锁的线程释放锁时，会通过futex唤醒等待队列中的一个线程，被唤醒的线程会重新竞争锁。这种机制减少了线程在用户空间的无效循环等待（只有发生竞争时才需要内核调用），提高了系统的效率 。

条件变量也是常用的线程同步原语，通常与互斥锁配合使用，用于线程间的通信和协作。其内核实现中同样依赖于等待队列和信号机制，当一个线程调用pthread_cond_wait等待某个条件成立时，它会先释放持有的互斥锁，然后将自己放入条件变量的等待队列中，并进入睡眠状态。当另一个线程满足条件后，它会调用pthread_cond_signal或pthread_cond_broadcast唤醒等待队列中的一个或多个线程。被唤醒的线程会重新获取互斥锁，然后检查条件是否满足，如果满足则继续执行，否则再次等待。

CPU流水线

亚当斯密在《国富论》中描述了这样一种场景，制作一枚回形针一般需要18个步骤，工厂里的工人平均每天也只能做100枚回形针，后来改进工艺，将制作流程分为18道工艺，然后让10名工人平均每人负责1～2道工艺，最后这10名工人每天可以制造出48000枚针，生产效率整整提高了几十倍。

流水线是工业社会化大生产背景下的产物，其实你我也只是社会这个大流水线上工人，负责自己的那一两道工艺，然后不断精进，并与他人协作，最终提高了整体的社会生产效率。

经典的CPU执行指令一般分为5个步骤：取指、译码、执行、访问内存、写回。假设每个步骤的执行需要一个时钟周期，那么一条指令执行完需要5个时钟，但是在这5个时钟中，每个步骤只执行了一次，其它4个时钟都在空闲。解决办法就是引入流水线，让所有步骤不停的执行。

流水线其实就是拿空间换时间，将每条指令分解为多个步骤，每一个步骤都有独立的电路单元执行，然后让不同指令的各个小步骤操作重叠，通过多条指令的并行执行，提升程序的整体运行效率，就像富士康工厂里的流水线一样，源源不断的执行一条条指令。

只要将步骤拆得足够细，每个步骤的完成时间非常短，那么就能提高流水线的吞吐能力，也就能提高CPU的主频（步骤越小，时钟就可以做到越短，比如时钟是2ns，那么CPU主频就是500MHz）。但流水线越深，电路会越复杂，芯片面积也就越大，功耗也会上升，这对于很多靠电池续命的移动设备是难以接受的，所以现实生产中往往需要在性能和功耗之间做一个平衡。

超线程&多核

超线程技术可以用厨师做菜来比喻，比如现在要做两道菜，你可以同时让两个厨师烧菜，或者给一个厨师配两套厨具，因为每个菜的制作过程中都有等待时间，然后厨师可以利用等待时间做另一个菜。在CPU内部，也有很多资源也是可以共享的，比如ALU、FPU、Cache等，有的则是线程独享的，比如寄存器状态、堆栈等。然后通过增加一些逻辑控制电路，就可以在一个物理Core上面实现两个逻辑Core了（类似给厨师配两套厨具）。

但是这两个逻辑Core并不能真正的并行干活，因为共享资源只有一份。而好处是同一个物理Core上两个逻辑线程的上下文切换代价很小，一般只需一个时钟周期就行，而不同处理器上的两个线程切换通常需要数百上千时钟周期。

  • 多核处理器

多核处理器可以让多个任务真正的同时执行，但是在一个CPU内部集成多个Core，需要考虑各个Core之间怎么连接，Cache如何共享等问题。可以通过总线或片上网络连接到同一内存，使它们能快速访问共享内存中的数据，但多个Core同时访问内存会出现缓存一致性问题，即不同Core的缓存中可能存在同一内存地址数据的不同副本。常见的缓存一致性协议有MESI，它会时刻监督着各个Core缓存中的数据状态，当一个Core修改了缓存中的数据，它会及时通知其他核心将对应的缓存数据标记为无效，确保所有核心看到的内存数据始终是一致的。

指令重排也是多核环境下的一个“陷阱”，在单线程环境下，指令重排不会影响程序的正确性，因为单线程的执行顺序是确定的。但在多核并发环境中，指令重排可能会导致不同核心上的线程看到不一致的执行结果，假设线程1先对变量a进行赋值，然后对变量b进行赋值，线程2需要先读取变量a的值，再读取变量b的值。由于指令重排，线程1中对变量b的赋值操作可能会先于对变量a的赋值操作被执行，这对线程1没有影响，但是线程2读取到的变量a和b的值就可能不符合预期，从而引发程序错误 。

如果想精准约束指令执行顺序、保障多核内存访问的一致性，就必须依靠内核提供的内存屏障机制，它也是解决多核乱序、支撑锁与原子操作正常工作的核心底层手段。内存屏障是一种特殊的指令，它可以阻止CPU和编译器对特定指令进行重排序，从而保证内存操作的顺序性和可见性。内存屏障会强制规定：屏障之前的内存访问操作必须完成并生效后，方可执行屏障之后的内存访问操作。它保证了在多线程环境下，不同线程对共享内存的访问能够按照预期顺序执行，避免产生数据竞争和其他并发问题。

操作系统线程

相对于硬件CPU中物理线程，比如常说的”4核8线程”，操作系统创建的线程可以认为是逻辑线程，说白就是内存中描述线程的一个结构体，系统可以创建成百上千个线程，然后对它们进行调度，也就是挂载到真正的物理线程上执行。

内核调度不同线程产生的开销一般称为上下文切换，当内核决定切换线程时，首先会将当前线程的寄存器值、程序计数器和栈指针的值保存到该线程的内核栈中，然后从调度队列中选择下一个要执行的线程，从新线程的内核栈中恢复寄存器的值、程序计数器和栈指针，最后CPU开始执行新线程的代码。

由于上下文切换需要保存和恢复大量的寄存器信息，并且切换后新线程访问的数据可能不在CPU缓存中，需要从内存中重新读取数据，从而降低系统的性能。因此，在设计程序时，应该尽量减少不必要的上下文切换，提高系统的整体性能。

  • 线程状态

主要关心就绪、运行、阻塞三个状态，线程创建完成就可以进入就绪队列等待被调度了，如果被调度分配到物理线程那么进入运行状态，等时间片跑完再放入就绪队列等待下次调度，如果在运行状态时发现需要等待资源或某个条件成立，那么就将其放入等待队列（具体等待队列有很多，比如sleep定时器队列，mute互斥锁队列，某个I/O块设备的等待队列等），当条件满足后再将其移入就绪队列。

Java线程

Java线程Thread是对不同操作系统的线程（比如Linux的pthread，或Windows线程）又做了一层统一封装，是JVM用于操作线程的一个代理对象，它的很多方法都是Native的。创建一个Thread就是创建一个普通Java对象，只有在start()时才会真正进行Native调用创建线程。

Java中的Thread也定义了6种状态，不过它的状态对应的是JVM中线程的状态，与底层操作系统线程的状态并不完全对应，比如它的Runnable状态就包括了就绪Ready和运行Running，并且就算线程处于I/O等待事件中时，对JVM线程来说依然是Runnable状态，因为它并不关心也没法关心底层线程在做什么，它只知道自己调用了什么Api。基于不同的Api动作，它在Blocked阻塞状态外，还新增了等待和限时等待两种状态。

  • 线程池

对于大批量任务的场景，一般都会使用线程池来应对，其思路就是创建一定数量的工作线程，然后将待执行的任务放入队列，然后让工作线程不停从队列中获取任务执行，这样可以避免频繁创建线程的开销。但是需要考虑工作线程数、和任务队列长度的设置问题。

如果线程数设置过大那么可能导致上下文切换的开销变高，而如果线程数设置过少，可能导致队列中的任务不能及时得到执行（比如已经执行的任务让线程都处于阻塞状态，没有更多的线程来执行队列中的任务）。所以根据经验，一般对于CPU密集型的任务会将线程数设置少一点，而I/O密集型任务会将线程数设置大一点。

虚拟线程

担心上下文切换开销高，主要还是因为Java只能创建线程，但没有调度的权力，，操作系统也不可能对外暴露这种Api。所以Java在jdk 19版本开始引入虚拟线程，它的思路是自己在用户态创建M个线程，然后映射到N个系统线程行执行，这些用户态线程可以由自己调度，调度开销只需拷贝几KB的数据，包括程序计数器、栈指针和少量寄存器，相比系统线程切换能快几百上千倍。

可以将虚拟线程看成一个更加复杂的线程池，只是队列中的不是任务，而是就绪的虚拟线程，这些虚拟线程相互之间可能有相互依赖或竞争关系，而且虚拟线程是复用的，所以相比简单的获取执行一次性任务要复杂得多，所以Java中选择了用ForkJoinPool来作为虚拟线程调度器的实现。

Go语言中这种技术称为协程，使用已经非常流行，其实Java也是迫不得已，只能追赶，否则在高并发程序设计场景中会越来越无法与Go相抗衡，Go语言在高I/O并发的程序中，能以极低的成本支撑百万级并发，而且Go的协程具有“从零设计”的先天优势, 没有历史包袱。

  • 示例

假设要执行10000个请求，按照普通java线程池的思路，可以创建一个线程数为10的线程池，然后将10000个请求抽象成任务放入队列，但这样的问题是如果先执行的10个请求由于I/0阻塞把线程占住，就会导致其它的请求任务只能在队列中干等，而此时工作线程实际上并没有工作。

如果将调度的对象由任务抽象改进成虚拟线程，那么如果已经在执行的虚拟线程被I/O阻塞，可以让出工作线程，切换成队列中的其它虚拟线程来执行，这样就提高了工作线程的利用率，并且过程不会涉及到系统线程上下文的切换，难点只在于对于虚拟线程的调度实现上，但这个对应用开发者完全透明。
