NIO Channel & Selector

words: 2.1k    views:    time: 7min
Java 基础
I/O

Channel 与之前的 流Stream 是差不多一个等级的，只不过 Stream 都是单向的，比如InputStream或OutputStream，而 Channel 可以是单向的，也可以是双向的，既可以同时支持读写操作，具体由其实现的接口所决定

可以看到 read 和 write 方法接受的都是一个 ByteBuffer 参数，其中 read 就是 Channel 往 ByteBuffer 中 put 数据，而 write 就是 Channel 从 ByteBuffer 中 get 数据，以便发送给其他远程主机。两者均返回已传输的字节数，可能比缓冲区的字节数少甚至可能为零，缓冲区的position位置也会前移对应的字节数，如果只进行了部分传输，缓冲区可以被重新提交给通道并从上次中断的地方继续传输，该过程可以重复进行直到缓冲区的hasRemaining()方法返回 false 值。

Channel

上图是 Channel 的类继承机构，类似于I/O的分类，通道也可以分为文件file通道，和套接字socket通道。

Socket 通道

在 socket 通道类中，DatagramChannel 和 SocketChannel 实现读和写接口，而 ServerSocketChannel 不实现，它负责监听传入的连接和创建新的 SocketChannel 对象，本身不传输数据。

它们在创建时都会创建一个对应的 socket 对象，即 Socket、 ServerSocket 和 DatagramSocket，可以通过socket()方法获取其关联的socket对象。反过来通过 socket 也可以通过getChannel()获取其对应的通道，但前提是这个socket对象必须是由通道类创建的，而不是直接new的。

ServerSocketChannel

ServerSocketChannel 是一个基于通道的 socket 监听器，其任务与 java.net.ServerSocket 相同，只是增加了通道语义，能够在非阻塞模式下运行。

通过静态工厂open()可以创建一个新的 ServerSocketChannel 实例，但是它没有绑定端口，需要获取其对应的 ServerSocket 进行端口绑定以及一些其它一些设置

如果在 ServerSocket 上调用accept()，那么总是阻塞直到返回一个 Socket 对象，如果在 ServerSocketChannel 上调用accept()，则会返回一个 SocketChannel，但它支持非阻塞模式运行，即如果没有连接接入，accept()会立即返回 null。

SocketChannel

SocketChannel 作用与 Socket 一样，用于封装点对点、有序的网络连接，即 TCP/IP。它们负责发起一个到监听服务的连接，直到连接成功，才能收到数据并且只会从连接到的地址接收。

类似的，通过静态工厂open()可以创建一个 SocketChannel，并且每个 SocketChannel 在创建时都会与一个 java.net.Socket 关联。只是 SocketChannel 提供了非阻塞创建连接的能力，也就是不等连接建立完成就立即返回，然后调用线程继续干别的事情，并将其注册到选择器上，然后等连接建立成功了，再进行对应的处理。

Selector

Selector 是NIO能提高服务并发能力的关键，它使得应用可以用单个线程来监听多个连接的事件，比如连接建立，数据到达等。它其实是操作系统提供的一个轮询服务接口，这样由操作系统统一负责对多个sockect连接进行监听。当发现有Socket准备好之后再返回应用线程，应用线程在拿到对应的 SocketChannel 后再通过 Buffer 进行数据读取和响应写回。

  • 创建

Selector 也提供了对应的静态工厂方法open()来负责创建，其向 SPI 发出请求，通过默认的 SelectorProvider 获取一个新的实例。

  • 注册通道

通道是支持非阻塞模式的，那么就需要一种方式来知道通道在什么时候操作就绪，而方式就是将其注册到 Selector 上。

要注意的是，在将通道注册到 Selector 上时，其必须处于非阻塞模式，否则将抛出 IllegalBlockingModeException 异常，并且一旦被注册，就不能再回到阻塞模式。

另外，在注册时需要指定感兴趣的事件 SelectonKey，其预定义了以下4种事件

一个通道可以被注册到多个选择器上，但对每个选择器而言，最好只注册一次。如果一个 Selector 上多次注册同一个 Channel，返回的 SelectionKey 总是同一个实例，即后面注册的感兴趣的操作类型会覆盖之前的。如果一个 channel 在不同的 selector 上注册，那么每次返回的 selectorKey 都是一个不同的实例。

  • 选择

每一个 Selector 都维护三个键集合

已注册的键集合(Registered key set)：与选择器关联的已经注册的键集合，并不是所有注册过的键都仍然有效。这个集合可以通过keys()方法返回，并且可能是空的。但这个键集合不可以直接修改，试图这么做的话会抛出 UnsupportedOperationException 异常。

已选择的键的集合(Selected key set)：已注册键集合的子集，通过selectedKeys()方法返回，其中每个键对应的通道都被选择器（在选择时）判断为已就绪的。

已取消的键的集合(Cancelled key set)：已注册键集合的子集，包含了调用过cancel()方法的键，这些键已经被无效化，但它们还没有被注销。该集合为选择器私有，无法直接访问。

对于选择操作，Selector提供了三种方式：

选择是累积的，一旦选择器将一个键添加到它的已选择的键集合中，它就不会再移除这个键。并且，一旦一个键处于已选择的键集合中，这个键的 readyOPS 将只会被设置，而不会被清理。这样就将键的管理操作交给了开发人员，比如移除那些已处理过的键，或者重新注册。

  • 停止选择

有三种方式可以唤醒阻塞在select()方法上的线程

wakeup()：使得选择器上的第一个还没有返回的选择操作立即返回，如果当前没有在进行中的选择，那么下一次对 select()方法的调用将立即返回，而后续的选择操作则正常进行。

close()：使得所有在选择操作中阻塞的线程都被唤醒，就像wakeup()方法被调用了一样。另外，与选择器相关的通道将会被注销，而键将被取消，注意这里并没有说会关闭通道连接。

interrupt()：即中断线程的语义，这里不再赘述。

  • 线程安全

要注意的是，选择器是线程安全的，但它们包含的键集合不是。
