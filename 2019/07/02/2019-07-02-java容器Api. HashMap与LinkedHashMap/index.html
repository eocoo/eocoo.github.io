java容器Api[1.8].HashMap与LinkedHashMap

words: 6.7k     views:

引言

1. 概述

java中，所有对象都会实现一个接口hashCode()，返回对象的hash。特点是对于任意一个给定的对象，它的值是明确的；而对于任意两个不同的对象，它们的值相同的概率非常小。 因此，如果定义一个长度为hash最大值（即int最大值）的数组，然后用对象的hash作为下标将其放入数组，另外对于少数hash相同的对象（称为hash碰撞）使用链表或者树进行链接， 那么我们基本上就可以根据对象的hash直接从数组中进行获取了，少部分可能要再次遍历下链表或者树。而这里的数组其实就是一个散列表，并且由于对象的地址就是计算的hash，因此很多人也称为hash表。

但是通常情况下，我们需要保存的对象个数不会那么多，如果数组长度定义成int最大值就浪费空间了，因此可以将长度定义成要保存的对象的个数，然后用对象的hash对数组长度进行取余， 就能得到其在数组中一个确定的位置。但是，如果数组长度缩小了，那hash碰撞的概率就增加了。
办法是提供一个系数，使对象个数与数组长度始终保持一个比例，这样就可以在空间使用率和碰撞概率两个问题之间进行权衡，比如HashMap中取了一个折中的值0.75。 另外，如果数组长度相对于hash值小得多，那么取余将变成就是取hash的最后几位，这很可能会丢失hash的散列特性，因此可以将hash值的高低位进行异或以中和其在低位上的散列程度。 以上其实就是HashMap的思路，它本质上就是一个数组实现的散列表，因此也是无序的。

LinkedHashMap其实就是在HashMap的基础上加了一个链表，为了维护对象的插入顺序，它只通过少量额外的操作，便达到了目的，并且保持HashMap的特性。

2. HashMap

2.1. 属性

2.1.1. table

table就是一个数组，在内部当作散列表来使用。其元素为Node，可以以链表或者红黑树的形式向下继续链接（当链表长度大于8且table的长度大于64时，会将链表转成红黑树）。

TreeNode继承自Node，加了一些实现红黑树时的必要属性。对于红黑树，其节点是有序的，因此在插入时必须要对节点进行比较，它在比较时首先检查Key是否为Comparable，如果是就可以直接比较， 否则根据System.identityHashCode(key)比较（虽然hash对数组取余相等，但不代表hash自身是相等的，如果真的相等就当小于处理）。红黑树在HashMap中并不是重点， 只是用来降低hash碰撞时的比较次数的一个手段，HashMap的初衷肯定是期望没有碰撞，这样从集合中获取多谢的时间复杂度就稳定为o(1)，更多红黑树的细节，可以看之前整理的文章：

2.1.2. loadFactor

理想情况下，肯定是希望所有的元素尽量的分散，最好是n个元素平均分布在长度为n的table上。但现实情况中，总免不了计算出的位置发生碰撞，如果碰撞了，就通过链表或者红黑树来将元素向后链接， 因此，在计算出散列表中的位置后，可能还要对链接的元素进行遍历，并进行hash比较和equals操作（同一个位置下的元素key的hash只是对表长度的取余相等，并不是hash相等）。
所以HashMap给了一个负载因子loadFactor，其实际存放的元素总数不能超过table.length * loadFactor，记为threshold，当元素总数达到threshold时就对table.length进行翻倍。 那么问题是负载因子的值设为多少合适呢，这时，就需要做一个权衡，如果负载因子比较小，那么碰撞的概率肯定低了，但空间消耗高了；反之空间利用率肯定高了，但碰撞概率也高了，

api中给了一个比较折中的值0.75：

As a general rule, the default load factor (.75) offers a good tradeoff between time and
space costs. Higher values decrease the space overhead but increase the lookup cost
(reflected in most of the operations of the HashMap class, including get and put). The
expected number of entries in the map and its load factor should be taken into account
when setting its initial capacity, so as to minimize the number of rehash operations. If the
initial capacity is greater than the maximum number of entries divided by the load factor,
no rehash operations will ever occur.

意思也差不多：作为一般规则，默认负载因子（0.75）在时间和空间成本上提供了很好的折衷。较高的值会降低空间开销，但提高查找成本（体现在大多数的HashMap类的操作，包括get和put）。 设置初始大小时，应该考虑预计的entry数在map及其负载系数，并且尽量减少rehash操作的次数。如果初始容量大于最大条目数除以负载因子，rehash操作将不会发生。 网上也有关于取值0.75的数学方法讨论，有兴趣可以看下HashMap的loadFactor为什么是0.75。

2.3. 静态工具

2.3.1. tableSizeFor

tableSizeFor方法是用来设置HashMap中散列表的长度，它确保返回一个大于且靠近给定数的2的幂，具体为什么这样做，后面会说明。 这里实现得非常巧妙，它通过5次无符号右移以及位或运算，将任意一个以二进制表示的数的最高位往右至多32位全部置为1， 然后再加1，这样使得所有位都进1，最后刚好得到一个比给定数高1位的2的幂。用等比求和也好说明：(1 + 2 + 2^2 + … + 2^n) + 1 = 2^(n+1), n <= 32。
并且运算之前会首先将给定数减1，以确保结果不会超过给定数。如果给定数刚好为2^x，那么减1之后最高位将为2^(x-1)，这样运算后进位的结果依然为2^x， 而如果给定数为2^(x-1) + m, 1 <= m < 2^(x-1), 结果肯定也是2^x。另外，考虑到int有效值区间，方法设置了返回值的上下界：2^0 ~ 2^30。

2.3.2. hash

hash方法用来给出key的hash值，后面会根据这个hash值来确定key在table中的位置。一个简单的办法是用table.length对key的hash值进行取余，不过源码中为了运算效率， 使用的位运算：(table.length - 1) & hash。这样的运算效果可以等价于hash % table.length，但前提是table.length必须为2的幂，这样table.length - 1才能确保低位全部为1， 然后进行位与运算得出key.hash在低位的值，这也正是前面tableSizeFor的用意。

并且，hash方法在返回key的hash值之前，会先将它hash值的高位与低位进行异或。由于table的长度一般会远小于key的hash值（table长度默认为2^4，也就是只有4位）。如果直接进行取余， 那么相当于只有hash的低位参与了运算，也就是说key在表中的位置就是它hash值的低位，那这样发生碰撞的概率还是比较高的。因为，虽然hash值的散列特性很好，但它的区间很大，就是int取值范围， 它并不会保证在低位上的散列。试想如果给定n个key，计算得出的hash值在低位完全相同，而高位全部不同，那么对于hash函数来说，它的目的已经达到了，但此时如果计算它们在table中的位置， 将全部碰撞在一起。因此，设计者考虑将hash的高位与低位先进行异或，这样可以中和一下高位和低位的散列特性，使计算的位置更加均匀，而且这样做不会增加多少性能开销。

另外，可以看出它对值为null的key的处理，固定返回hash值为0，因此最终只能存在一个key为null的元素，并且与其他hash取余为0的元素一起放在散列表的第0位。

2.4. 接口实现

从类图中可以看到，HashMap实现的接口有很多，不过这里就说一个比较有代表性的put实现，其他接口实现大体类似。 另外，说一下HashMap中迭代器的实现，虽然HashMap自身没有实现Iterator接口，但是它通过内部的entrySet来帮自己实现。

2.4.1. put

先说下大概思路，首先根据hash计算在table中对应的位置，如果对应位置为空，那么就新建一个节点；否则，就依次与位置下的节点进行比较，如果存在相同的就直接替换，否则继续向下链接。 如果原来为链表，且链接后发现链表长度大于8，那么会对散列表表的长度进行扩充或者将链表转化成红黑树。
最后，如果发生了新增，会在最后检查元素的总数是否超过了阈值threshold，如果超过了，就需要对表的长度进行扩充，即创建一个两倍长度的散列表，并将元素放到新的表中， 这里也是实现得非常巧妙，它将位置的计算转换为判断一个二进制位是0还是1的问题，具体可以看下面它的实现。

贴一下resize()方法，它负责对列表的长度以及阈值threshold进行初始化和扩充调整

再贴一下treeifyBin(tab, hash)方法，如果链表新增之后长度大于等于8，那么会扩充表的长度，或者将对应的链表转化为红黑树。

2.4.2. entrySet

在属性中有一项entrySet，在调用entrySet()时就返回它，但HashMap中的状态一直在变化，怎么能保证返回的entrySet跟上变化呢。事实上，它的类定义没有任何状态， 而是完全引用外部类HashMap的状态，因此HashMap的状态就是它的状态。

对HashMap的遍历其实是对entrySet的遍历，因为HashMap没有实现Iterator接口，因此无法直接遍历，但entrySet可以，它返回一个EntryIterator<Map.Entry<K,V>>实例。

3. LinkedHashMap

LinkedHashMap直接继承自HashMap，除了迭代器，它所有的操作都直接复用的HashMap的实现，只是在过程中加了一个链表操作。 现在可以理解为什么HashMap.TreeNode要绕个弯继承一下LinkedHashMap.Entry了，因为这样LinkedHashMap在复用HashMap的红黑树操作时就可以同时维护自己的链表了。
