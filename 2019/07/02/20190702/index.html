基于spring构建计划任务 设计文档

words: 3.7k    views:    time: 13min
代码笔记
schedule

https://github.com/shanhm1991/spring-fom

写这个工具一开始是为了解决各种离线数据文件的处理，其中涉及到一些定时任务，但除了定时之外，还希望能支持任务批处理，以及能对任务进行实时监控和一些干预等操作。 如果直接使用 spring 提供的 Schedule，免不了要做一些额外的工作，于是自己尝试设计和实现了这样一个计划任务框架，由于当时是设计用来处理文件的，又是基于 spring 来实现的， 所以取名为 spring-fom。

1. 概要设计

如果概括起来，思路也比较简单。主要的任务调度逻辑都封装在ScheduleContext中，其维护了一个定时线程和一个线程池，另外自定义了一套状态转换机制。 这样每个ScheduleContext就相当于一个独立的任务调度器，其生命周期，比如加载、启动、终止，完全基于 spring 的应用上下文。 对于具体的任务则抽象为 Task，其实就是在 Callable 的基础上定义了一套任务执行模板。因此，整个 spring-fom 的功能，都是围绕 ScheduleContext 和 Task 实现的。

1.1. 功能设计

功能设计也可以理解成需求定义，就是希望spring-fom能支持的一些特性。当然除了下面列出的这些主要特性之外，还需要将模块和代码结构设计的足够清晰，以便能够灵活扩展和定制。 只是这里主要还是从功能切入，解释一下实现思路，对于具体的代码技巧，比如如何利用spring上下文管理生命周期，如何利用注解和动态代理配合接口使用，以及如何保证线程安全等细节就不再赘述。

1. 支持最基本的定时任务，以及三种定时语义（cron/fixedRate/fixedDelay）
2. 支持定时多任务执行
    实现接口ScheduleFactory可以创建批任务，或者通过@Scheduled指定多个任务方法
    实现接口ScheduleCompleter可以定义任务全部完成时的处理
3. 支持实时启动 / 关闭定时任务，即能够在不重启应用的情况下，启动或停止指定的定时任务
    实现接口TaskCancelHandler可以自定义在关闭时任务的取消处理，默认通过Interrupt中断
    实现接口ScheduleTerminator可以自定义定时器关闭时的处理
4. 支持手动干预定时任务的执行，比如当定时任务未到执行时机时，可以手动使其立即执行
    如果当前任务正在执行则能够支持两种策略：直接忽略，或者等待本次执行完成后立即再重新执行
5. 支持实时修改任务配置，并支持持久化，这样重启后修改依然有效
6. 支持任务超时检测和处理，通过taskOverTime可以设置任务超时时间
    同样实现接口TaskCancelHandler也可以自定义任务超时后的取消处理，默认通过Interrupt中断
    通过detectTimeoutOnEachTask可以设置是否对每个任务单独检测超时，有可能在多任务场景中希望对整体任务计算超时
7. 支持任务冲突检测
    通过enableTaskConflict可以打开任务冲突检测，即如果提交任务时，发现已经存在对应id的任务，并且还在运行，则忽略本次任务， 这在文件处理的场景中比较有用；
8. 支持一些运维管理操作的接口
    提供了一个简单的运维界面，可以通过开关enableFomView打开

1.3. 状态定义

对于ScheduleContext中维护的几个状态，其状态转换机制可以如下图所示：

  • 如果处于状态：INITED 或 STOPPED

可以接收外部线程的startup，然后启动定时线程，由定时线程将状态切换为 RUNNING；

  • 如果处于状态：RUNNING

正常情况下，定时线程会等待任务结束后，将状态切换为 SLEEPING，如果没有设置定时计划(一次性任务)，则切换为 INITED；

另外，可以接收外部线程的shutDown，由外部线程将状态切换为 STOPPING，并请求中断定时线程。然后定时线程会跳过等待任务结束的过程，并很快检测到 STOPPING 状态， 接着尝试取消还在执行的任务，关闭线程池，并在所有任务真正结束后将状态切换为 STOPPED；

也可以接收外部线程的startup，但默认是忽略，如果将ignoreExecRequestWhenRunning设置为false，那么会在本次任务结束之后会立即再重新执行一次；

  • 如果处于状态：SLEEPING

正常情况下，定时线程会在sleep结束后进行下一次任务执行，并将状态切换为 RUNNING；

另外，可以接收外部线程的execNow，由外部线程中断定时线程的等待，然后定时线程立即开始下一次任务执行，并将状态切换为 RUNNING；

也可以接收外部线程的shutDown，也是由外部线程中断定时线程的等待，但在这之前，外部线程会将状态切换为 STOPPING， 然后定时线程在开始下一次任务之前会检测到这个 STOPPING 状态，接着定时线程会关闭线程池，将状态切换为 STOPPED；

  • 如果处于状态：STOPPING

此时忽略一切外部请求，直到定时线程等待所有任务结束之后，由定时线程将状态切换为 STOPPED；

2. 详细设计

除了定时线程和状态之外，每个ScheduleContext还维护了一个私有的线程池，其对于具体的任务就是委托给线程池来执行的，其实ScheduleContext中的状态基本上就是其线程池状态的体现。 下面就具体说明下ScheduleContext将任务委托给线程池执行之后，是如何检测任务结束以及超时的，至于其它的细节实现，根据上面的描述，基本上已经能了解大概，就不再赘述了。

2.1. 任务执行

对于任务的执行可以直接提交给线程池异步执行即可，问题是如何检测提交的任务在什么时候结束，尤其是提交了多个任务之后。 比较简单的想法可能是由定时线程在提交任务时记录一个提交任务数，然后由工作线程在任务执行结束时将计数减1，如果减完后计数为0，则表示任务全部结束。

但在实现中，会发现在提交时这个任务实际提交数无法直接确定，暂且不考虑提交异常的问题，如果要检测任务冲突，那只能遍历进行提交，每次提交时需要检测任务id是否已经存在并正在运行， 如果是则放弃本次任务，这样只能在每个任务实际提交成功后才能将提交任务数加1。

于是问题又变复杂了，就是在工作线程执行任务，或将任务计数减1的同时，定时线程可能也在提交新的任务，并将任务计数加1。

所以，首先要保证在检测任务全部结束时，提交任务这个动作已经结束，否则如果任务执行足够快，可能出现后面的任务还没来得及提交，工作线程已经将提交的任务全部执行完并将计数置为0， 从而误以为所有任务都已经执行完了。对于这个问题，可以加一个提交任务是否结束的标识即可，这样定时线程在提交结束后将这个标识置为真，然后工作线程在任务执行结束并将计数减1后， 除了判断计数是否为0，还需要判断提交是否已经结束。

这样看上去好像没什么问题了，但是在定时线程提交完任务，与定时线程将提交结束标识置为真，两个操作之间并没有同步。可能出现这样的场景：定时线程已经全部提交结束了， 并且工作线程也把任务全部执行结束了，但在工作线程判断任务是否全部结束时，定时线程还没将提交结束标识置为真，于是工作线程就直接结束了。这样如果希望在任务全部结束时触发执行一些处理， 那么可能就会错失了执行的时机。所以，在定时线程提交结束之后，也需要检测一下任务是否已经全部结束，这样如果工作线程错过了执行，可以由定时线程来作下把关。

2.1. 超时检测

上面ScheduleBatch提供了一个限时等待方法waitTaskCompleted(long taskOverTime)，如果任务设置了超时，那么在等待任务结束时将会使用限时等待。

超时检测的实现思路并不难想，主要是借助于延时队列DelayQueue，并通过Delayed来包装一下任务对应的future

这样当定时线程提交结束后，首先等一个给定的超时时间overTime，如果等完还有任务没有结束，那么获取这些任务的耗时。如果已经超时，则尝试取消；如果还没有超时，那么计算一下任务剩余的可用时间， 并封装成Delayed放入延时队列。

接下来就是反复从延时队列中获取一个时间最近的任务来判断，同样的如果超时了就取消，否则重新计算剩余时间再放回队列。这样如果最后队列为空，就表示所有的任务都已经结束或者超时。 要注意的是，这时并不代表任务都已真正结束，定时线程对于每个任务，检测到超时只会尝试一次取消，如果任务不响应，定时线程也没办法，可能还要通过scheduleBatch.waitTaskCompleted() 来进行最后的兜底。
