引言

前面主要梳理了BeanDefinition的解析注册，注册之后就是对BeanDefinition的使用了，即利用BeanDefinition生产bean。 这里还是以前面示例中的获取调用User user = (User)beanFactory.getBean("test")作为入口进行分析。

getBean

简单概括下下面的实现，首先是进行beanName的转换，因为spring支持别名和beanFactory的方式获取bean，所以需要先统一转换下，然后尝试从单例缓存中获取，如果没有获取到则进行创建， 不过这里创建根据BeanDefinition分为三种不同的策略，即单例、多例以及scope

1. transformedBeanName

获取时可能传的是别名，或者FactoryBean，需要先统一转换为真正的beanName，因为后面各种验证的逻辑需要有一个统一的确定的名称

去除FactoryBean的修饰符’&’

获取别名对应的真正的beanName

FactoryBean

通常spring可以根据的配置利用反射来实例化bean,但某些情况下，实例化bean的过程可能比较复杂，这时如果可以采用编码的方式也许能更加灵活和简单。为此，spring提供了工厂接口FactoryBean。 也就是说当的class属性是FactoryBean时，获取的bean将是getObject()的返回实例而不是FactoryBean本身。

2. getSingleton

2.1. getSingleton != null

2.1.1. getObjectForBeanInstance

如果从单例缓存中获取成功，那么在返回之前还需要做进一步的处理，因为这里获取到的有可能是FactoryBean

2.1.1.1. getObjectFromFactoryBean

上面实际上只是做了一些类型校验的工作，接下来交给FactoryBeanRegistrySupport，它负责调用factory.getObject()，并对结果进行缓存，另外这里在创建完bean之后调用了一个事件，即postProcessObjectFromFactoryBean， 并且通过缓存singletonsCurrentlyInCreation确保单例的事件只会触发一次（相关代码行：21,25,32）， 至于具体的实现则交给用户自定义

2.1.1.2. postProcessObjectFromFactoryBean

对于BeanPostProcessor具体在子类AbstractAutowireCapableBeanFactory，其实就获取用户自定义的事件然后依次调用

2.2. getSingleton == null

如果从单例缓存中没有获取到，那么接下来就要根据BeanDefinition的情况进行不同策略的创建了。当然在创建之前，还有些准备和校验工作。

2.2.1. isPrototypeCurrentlyInCreation

如果bean是多例，但是却存在循环依赖，则直接报错，这里检测循环依赖的想法很简单，就是每次之前先把beanName放到prototypesCurrentlyInCreation中，比如创建A之时触发创建B，但是创建B又反过来依赖A，那么这时将会发现A已经在创建的名单中了。

2.2.2. getParentBeanFactory

如果存在parentBeanFactory，并且自己这里已经注册的BeanDefinition没有对应的beanName，那么久委托给父工厂，想当于一个递归调用

2.2.3. markBeanAsCreated

在实际创建之前，首先将beanName记录到已经创建过的名单中去

2.2.4. getMergedLocalBeanDefinition

将GenericBeanDefinition转换为RootBeanDefinition，如果父类bean不为空的话，则合并父类的属性

2.2.5. getDependsOn

递归调用getBean初始化依赖的bean

2.2.6. createBean

上面的准备工作结束后，便是进行真正的创建，并且根据scope根据scope分成了三种情况，不过只是一些前后的工作不同，主要的创建过程都是复用的。

2.2.6.1. prepareMethodOverrides

对应前面BeanDefinition解析中的lookup-method和replace-method属性，这里只是简单的做了一下校验，如果对应的方法名称不存在则提前报错，如果对应名称的方法只有一个则将其标记为不需要通过参数类型匹配， 至于剩下的工作留在后面继续。

2.2.6.2. resolveBeforeInstantiation

这里在真正创建之前给了一个事件来做一些自定义处理，并且很重要的这里对处理之后的结果结果进行了判断，如果不为空，则直接返回，跳过后面的创建过程。对于AOP的实现主要就是基于这里提前返回的动态代理对象， 另外对于动态代理对象也尽量尝试调用其postProcessAfterInitialization事件
