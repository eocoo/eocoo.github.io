JUC FutureTask

前面整理线程池时说过Executor是执行任务的抽象，而Runnable是任务的抽象，但实际上在具体提交任务时，它会将任务统一适配成FutureTask，它是实现了接口RunnableFuture， 相当于是Runnable和Future的一个整合，但在实现中它又依赖于Callable，可以这样理解，它以Runnable的身份对外提供执行调用，然后在执行过程再委托给依赖的Callable处理， 另外它以Future的身份提供一些接口服务，如此，可以画出如下UML

1. 属性

1.1. state 状态

对于任务，设置了以上7种状态，其状态转换机制也如注释中所示，但如果不作细分的话，三个状态就可以区分任务的进度了

  • NEW：表示任务新建以及正在执行的过程
  • COMPLETING：表示任务执行完成或者异常结束，剩下的只是赋值而已
  • CANCELLED：任务在执行完成前取消了，至于中断状态只是用来描述取消之后中断任务的过程

所以，我们可以看到其对任务是否结束或者取消的判断非常简洁

1.2. waiters

waiters以栈的形式用来维护阻塞在get()上的等待线程，对于获取到结果或者等待超时的线程节点应该删除，这里并没有用什么标识位来标识，而是直接将thread置为null就表示自己可以删除

2. 构造器

构造器比较简单，就是初始化任务以及状态，并统一适配成callable

3. 接口

3.1. run

基本流程都写在注释中了，即让设置runner成功的线程来执行任务Callable，然后修改状态并设置结果，最后再修改状态并唤醒等待结果的线程，这里通过两个状态的修改操作，保证了结果赋值的线程安全

3.2. get

获取结果有三种情况，即正常结束，异常结束或者被取消。主要看下这里如何阻塞的，以及阻塞结束后如何删除的，具体注释中已经描述得很详细，这里就不赘述了

3.3. cancel

对于取消分了两种情况，区别在于是否对执行的任务进行中断，如果为false，就是直接唤醒在get()上阻塞的任务，并让其获得一个取消异常，至于任务还是照样执行，只是结果不要了。而如果为true， 则是在唤醒等待结果的线程之前先对执行的任务发出一个中断请求，至于是否得到响应，则看具体任务的实现。
